## Runtime model

The backend now uses one storage model:

- matching and orderbooks stay in memory
- runtime open orders and fills stay in memory only
- a local disk checkpoint is rewritten periodically

Checkpoint contents:

1. provisioned users and API keys
2. exchange controls and market definitions
3. balances, positions, and realized PnL
4. competition leaderboard snapshots

Config knobs:

- `CHECKPOINT_PATH`
- `CHECKPOINT_INTERVAL_SECONDS`

## Startup recovery

On process start, the backend:

1. loads the latest checkpoint file if one exists
2. restores users, markets, balances, positions, and competition snapshots
3. starts with empty orderbooks and no restored resting orders
4. expects clients to reconnect and resubscribe for fresh market-data snapshots

This means user positions and PnL survive a restart, but live books do not.

## What recovery does not do yet

- no full replay-based settlement recovery
- no historical recomputation of positions and PnL from an event journal alone
- no durable continuation of market-data sequence numbers
- no recovery of runtime-only open orders or fills
- no crash guarantee for changes made after the most recent checkpoint flush

## Operational meaning

Today, restart recovery is intentionally snapshot-based rather than replay-driven. It restores competition state, but every restart is effectively a fresh market-data session.
