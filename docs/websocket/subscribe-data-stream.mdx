---
title: "Subscribe Data Stream"
description: "Subscribe to the aggregated public market-data stream on the socket."
---

Send **`subscribe`** with **`"channel": "data"`** to receive one aggregated book snapshot, then sequenced price-level updates for a single market.

The public **`data`** stream is ideal when you want compact depth. The server rolls public order activity into **price levels** and flushes updates every **100ms**. No API key is required.

<RequestExample>

```json Client Message
{"op":"subscribe","channel":"data","market":"BTC-USD"}
```

</RequestExample>

<ResponseExample>

```json First Response
{
  "type": "snapshot",
  "channel": "data",
  "market": "BTC-USD",
  "sequence": 0,
  "bids": [
    { "price": 100, "quantity": 12 },
    { "price": 99, "quantity": 8 }
  ],
  "asks": [
    { "price": 101, "quantity": 6 },
    { "price": 102, "quantity": 10 }
  ]
}
```

</ResponseExample>

## Client fields

- `op` must be `subscribe`.
- `channel` must be `data`.
- `market` is the market you want to subscribe to.

## What happens next

- The server first sends a `snapshot`.
- `bids` and `asks` in that snapshot are aggregated price levels, not raw order rows.
- After the snapshot, the server sends `delta` messages with contiguous `start_sequence` to `sequence` ranges.
- The exchange batches those price-level updates per market and flushes them every `100ms`, so one `delta` can carry several ordered events.
- A `level_updated` delta sets the total visible quantity for one side and one price.
- `quantity: 0` means that price level should be removed from your local book.
- `trade` deltas carry executed price and quantity only.
- If the backend detects a gap or a lagged receiver, it sends `resync_required`.

## Notes

- The current implementation supports one market-data subscription per connection.
- If you need multiple markets, open multiple sockets.
- Use `start_sequence` to validate the first event in a batched `delta`, then store `sequence` as your new last applied market sequence.
- Use `order_state` messages for your own exact resting-order state. The public `data` stream is intentionally aggregated.
