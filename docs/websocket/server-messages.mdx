Below are the WebSocket payloads your client should recognize and handle cleanly.

## `heartbeat`

```json
{"type":"heartbeat"}
```

## `authenticated`

```json
{
  "type": "authenticated",
  "trader_id": "9d64e278-8b56-46e9-9cab-18c5b22f2fb9",
  "team_number": "TEAM-042"
}
```

## `snapshot`

```json
{
  "type": "snapshot",
  "channel": "data",
  "market": "BTC-USD",
  "sequence": 0,
  "bids": [
    { "price": 100, "quantity": 12 },
    { "price": 99, "quantity": 8 }
  ],
  "asks": [
    { "price": 101, "quantity": 6 },
    { "price": 102, "quantity": 10 }
  ]
}
```

`snapshot` is the aggregated `data` stream view. `bids` and `asks` are price levels, not per-order rows.

## `delta`

```json
{
  "type": "delta",
  "channel": "data",
  "market": "BTC-USD",
  "start_sequence": 1,
  "sequence": 1,
  "events": [
    {
      "kind": "level_updated",
      "side": "BUY",
      "price": 100,
      "quantity": 14
    },
    {
      "kind": "trade",
      "price": 101,
      "quantity": 1
    }
  ]
}
```

`start_sequence` is the sequence number of the first event in the batch, and `sequence` is the last applied sequence in that same batch.

`level_updated` sets the full visible quantity for one side and one price. `quantity: 0` means remove that level from your local book. These are price-level updates, not per-order add, amend, or remove events.

The exchange derives browser-facing market data from the canonical order-event stream, batches contiguous market-data events per market, and flushes them roughly every `100ms`, so a single `delta` can contain multiple ordered events.

## `fill`

```json
{
  "type": "fill",
  "fill": {
    "fill_id": "fef1b4fd-d0b0-4270-8831-b4c51aa43f1d",
    "market": "BTC-USD",
    "maker_order_id": "4d6c28f4-b349-4a4d-8927-8141c36eb6eb",
    "taker_order_id": "66377526-7b98-485a-8c40-7024e68fa3c5",
    "price": 100,
    "quantity": 1,
    "occurred_at": "2026-03-18T15:28:04.132958Z"
  }
}
```

## `order_state`

```json
{
  "type": "order_state",
  "order": {
    "id": "66377526-7b98-485a-8c40-7024e68fa3c5",
    "trader_id": "9d64e278-8b56-46e9-9cab-18c5b22f2fb9",
    "market": "BTC-USD",
    "side": "BUY",
    "price": 100,
    "quantity": 2,
    "remaining": 1,
    "created_at": "2026-03-18T15:28:01.342510Z"
  },
  "status": "open"
}
```

`status` is one of:

- `open`
- `filled`
- `canceled`

Use `order_state` for your own exact resting-order detail. This is the per-order account feed; the market-data `snapshot` and `delta` messages are intentionally aggregated.

For market orders, accepted taker flow is terminal. You should expect `status: "filled"` with `remaining: 0` because market orders never rest on the book. In that case, `order.price` is the weighted average executed price across the immediate fills, rounded down to an integer.

## `market_state`

```json
{
  "type": "market_state",
  "market": {
    "market_id": "BTC-USD",
    "display_name": "BTC-USD",
    "base_asset": "BTC",
    "quote_asset": "USD",
    "tick_size": 1,
    "min_order_quantity": 1,
    "min_price": 90000,
    "max_price": 110000,
    "reference_price": 100000,
    "settlement_price": null,
    "status": "disabled",
    "created_at": "2026-03-18T16:05:00.000000Z",
    "updated_at": "2026-03-18T16:05:00.000000Z"
  }
}
```

Treat `market_state` as the live websocket push for market metadata and lifecycle status. Apply it immediately so tabs, ticket gating, and other market-aware UI stays current without waiting for a REST refresh.

## `market_deleted`

```json
{
  "type": "market_deleted",
  "market_id": "BTC-USD"
}
```

Treat `market_deleted` as the authoritative signal that the market no longer exists in the active market list. Remove it from tabs, switch away if it was selected, and stop treating any cached metadata for that symbol as live.

## `admin_message`

```json
{
  "type": "admin_message",
  "message": {
    "message_id": "26aa26f5-8d20-4a6c-9b66-99549585036f",
    "target_username": null,
    "target_trader_id": null,
    "market": "BTC-USD",
    "level": "info",
    "title": "Desk notice",
    "body": "Trading will stop in five minutes.",
    "created_at": "2026-03-18T16:05:00.000000Z"
  }
}
```

Treat **`admin_message`** as guidance from organizersâ€”trading halts, settlement notes, or other announcements that may change how you should behave even before you refresh REST.

## `resync_required`

```json
{
  "type": "resync_required",
  "channel": "data",
  "market": "BTC-USD",
  "expected_sequence": 5,
  "current_sequence": 9,
  "reason": "market data lagged by 3 messages; resubscribe for a fresh snapshot"
}
```

Step-by-step recovery: [Recovery and resync](/reference/recovery-and-resync).

## Handling checklist

- **`heartbeat`** â€” connection is alive; you can ignore the body if you only need a liveness signal.
- **`authenticated`** â€” binds the socket to your private user stream and returns your `team_number`.
- **`snapshot` / `delta`** â€” aggregated public **`data`** stream; authoritative while your subscription is healthy.
- **`delta`** â€” confirm **`start_sequence`** continues from your last applied sequence, apply **`events`** in order, then save **`sequence`**.
- **`fill` / `order_state`** â€” your private activity after authentication.
- **`market_state`** â€” live changes to market definitions (`status`, ticks, etc.).
- **`market_deleted`** â€” remove the market from your live market list immediately.
- **`admin_message`** â€” organizer announcements; pair with REST refresh when in doubt.
- **`resync_required`** â€” drop and rebuild local state using the recovery guide.
