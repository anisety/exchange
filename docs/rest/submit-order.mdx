---
title: "Create Order"
api: "POST /api/v1/orders"
description: "Submit a new limit or market order."
---

Submit limit or market orders through REST when you need deterministic request/response behavior. Use the WebSocket order entry paths if you prefer to keep order actions and acknowledgements on a single low-latency socket.

<RequestExample>

```bash Limit request
curl \
  -X POST \
  -H "x-api-key: YOUR_API_KEY" \
  -H "content-type: application/json" \
  https://exchange.jamesxu.dev/api/v1/orders \
  -d '{"market":"BTC-USD","side":"BUY","order_type":"limit","price":100,"quantity":2}'
```

```bash Market request
curl \
  -X POST \
  -H "x-api-key: YOUR_API_KEY" \
  -H "content-type: application/json" \
  https://exchange.jamesxu.dev/api/v1/orders \
  -d '{"market":"BTC-USD","side":"BUY","order_type":"market","price":0,"quantity":2}'
```

</RequestExample>

## Headers

<ParamField header="x-api-key" type="string" required>
  Assigned competitor API key.
</ParamField>

## Body parameters

<ParamField body="market" type="string" required>
  Target market identifier, for example `BTC-USD`.
</ParamField>

<ParamField body="side" type="BUY | SELL" required>
  Order side.
</ParamField>

<ParamField body="price" type="u64" required>
  For `limit` orders, the limit price. It must be greater than zero and aligned to the market tick size. For `market` orders, this field is ignored; send `0`.
</ParamField>

<ParamField body="order_type" type="limit | market" default="limit">
  Order behavior. Omit this field or set it to `limit` to place a normal resting-capable limit order. Set it to `market` to consume opposite-side liquidity immediately without resting.
</ParamField>

<ParamField body="quantity" type="u64" required>
  Order quantity. Must be greater than zero and at least the market minimum quantity.
</ParamField>

## Response fields

<ResponseField name="order" type="object" required>
  Accepted order record after any immediate matching has been applied.
</ResponseField>

<ResponseField name="order.id" type="string" required>
  Stable order identifier used for amend and cancel requests.
</ResponseField>

<ResponseField name="order.remaining" type="u64" required>
  Quantity still resting after any immediate fills. This is always `0` for market orders.
</ResponseField>

<ResponseField name="order.price" type="u64" required>
  Submitted limit price for limit orders. For market orders, this is the weighted average executed price across immediate fills, rounded down to an integer.
</ResponseField>

<ResponseField name="fills" type="object[]" required>
  Immediate executions produced while the order was being matched. Market orders may return multiple fills if they sweep through multiple price levels.
</ResponseField>

<ResponseField name="resting" type="boolean" required>
  `true` when some quantity was posted to the book. `false` when nothing rests, including all market orders.
</ResponseField>

## Validation rules

- `order_type` defaults to `limit` when omitted.
- `price` must be greater than zero and aligned to the market `tick_size` for limit orders.
- If the market defines `min_price` or `max_price`, limit prices must stay within that inclusive band.
- Send `price: 0` for market orders. The server ignores the field.
- `quantity` must be greater than zero and at least the market `min_order_quantity`.
- The order must keep your worst-case per-market net position within `+/-1000` unless the API key belongs to an admin trader.
- You can sell from flat and go short.
- Crossing the book is allowed. Limit orders may fill immediately, partially rest, or fully rest depending on available liquidity.
- Market orders consume as much opposite-side liquidity as is currently available and never rest on the book.

## Common rejects

- `trading is currently disabled`
- `market is disabled`
- `market has already been settled`
- `price must align to tick size ...`
- `price must be at least ...`
- `price must be at most ...`
- `quantity must be at least ...`
- `market order could not be filled because no opposite-side liquidity is available`
- `projected net position for ... would be ...; limit is +/-1000`
