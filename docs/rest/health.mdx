---

Use **`GET /health`** to confirm the service is up and to see whether checkpoint writing or internal queues are under stress. You do not need an API key.

<RequestExample>

```bash Request
curl https://exchange.jamesxu.dev/health
```

</RequestExample>

## Response fields

<ResponseField name="status" type="string" required>
  Top-level exchange health. `ok` means the service is reachable and checkpoint persistence (if enabled) is not retrying, backpressured, or stopped, and `runtime_dispatch`, `account_dispatch`, and `persistence_dispatch` are not backpressured or stopped. `PersistenceMode::disabled` alone does not imply degraded.
</ResponseField>

<ResponseField name="service" type="string" required>
  Static service identifier. Currently `exchange`.
</ResponseField>

<ResponseField name="now" type="string<date-time>" required>
  Current server timestamp.
</ResponseField>

<ResponseField name="persistence" type="object" required>
  When disk checkpointing is enabled, checkpoint-writer health and flush telemetry (in-memory storage with periodic JSON checkpoint writes). When checkpointing is disabled, this reflects the in-memory repository persistence view (`mode` is typically `disabled`).
</ResponseField>

<ResponseField name="runtime_dispatch" type="object" required>
  Queue telemetry for market-data and runtime fanout work.
</ResponseField>

<ResponseField name="account_dispatch" type="object" required>
  Queue telemetry for account read-model updates.
</ResponseField>

<ResponseField name="persistence_dispatch" type="object" required>
  Queue telemetry for ordered write-side persistence work.
</ResponseField>

<ResponseField name="account_barrier" type="object" required>
  Barrier wait telemetry for submit, cancel, and amend paths.
</ResponseField>

## What to watch

- `status = "degraded"` means the checkpoint writer is retrying, backpressured, or stopped, or at least one of `runtime_dispatch`, `account_dispatch`, or `persistence_dispatch` is backpressured or stopped.
- `persistence.backlog_depth` shows how far durable writes are trailing the in-memory exchange state.
- `runtime_dispatch`, `account_dispatch`, and `persistence_dispatch` expose queue depth and high-water marks for the off-thread architecture.
- `account_barrier.submit.max_wait_time_ms` tells you how long requests have recently waited for read-your-writes account state to catch up.
